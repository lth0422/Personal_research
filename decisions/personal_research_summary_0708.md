## 1. 연구 주제 (변경 없음, 한 문장)

> **실시간 진동 결함 진단 태스크의 윈도우 크기·진단 주기·모델을,
> 기계 상태와 시스템 여유에 따라 runtime에 조절하는 elastic scheduling 기법**

병목: W, H, M이 설계 시점에 고정 → 정밀 고정이면 평시 낭비·과부하 시 deadline miss, 경량 고정이면 이상 시 감지 지연. "기계는 대부분 정상인데 시스템은 항상 한 가지 가정으로 돈다"는 비효율.

## 3. 핵심 개념 (0702~0707 정리 유지 — 면담에서 검증됨)

면담에서 실제로 사용되어 유효성이 확인된 정리들:

- **W/H/M 정의와 T = H/fs**: 교수님께 "비중첩 슬라이스라 주기가 윈도우 크기로 결정됐고, 주기를 늘리려면 hop size로 조절" 설명 → 통과. "그게 문제 정식화가 되게 중요한 거거든요" 확인받음.
- **mode = (W,H,M), (W,M)별 사전 학습 인스턴스**: "모델 자체를 다르게 하겠다" 설명 → 교수님이 imprecise computation(mandatory/optional), mode change(페이즈 전환 시 태스크셋 변경) 등 고전 개념과 연결해주심 → **서베이에서 이 고전 계보를 반드시 커버할 것** (이미 아카이빙됨: Imprecise Computations 논문).
- **C ∝ W² (4배 축소 시 11.4배)**: 교수님이 직접 인용하며 utilization 계산 과제 제시. 단 "그건 CMSIS 기준이었고 모델 내부 구조에 따라 달라진다" — 모델별로 C(W) 곡선이 다름을 명시할 것.
- online/offline 구분, anomaly score 옵션, (q,S) 사분면 시나리오 — 0707 정리 그대로 유효 (부록으로 유지).

## 4. 서베이 지침 (피드백 3·6·7·10 — 교수님 지시 그대로)

> 원칙: "단순히 참고문헌을 넣는 게 아니고, 그 연구들의 **접근 방법들을 다 싹 정리**해야 돼."

### 산출물 1 — Real-Time Fault Diagnosis 분류표

- 키워드: "real-time fault diagnosis" 및 인접 (real-time bearing/machine fault, on-device fault diagnosis)
- 범위: **최근 5년 이내**
- 분류 축 (교수님 명시):

| 논문 | 연도 | RTOS 유무 | 실시간성 접근 | 플랫폼 | 진짜 실시간? |
| --- | --- | --- | --- | --- | --- |
| (예) TinyML bearing | 2025 | 없음 | 경량 모델만 | MCU | best-effort (근실시간) |
| (예) 우리 KCC | 2026 | Zephyr | RTOS + 측정 | STM32F407 | deadline 기반 |

- 핵심 구분: RTOS 없이 TinyML 적용·양자화·프루닝으로 실행시간만 줄인 것 = **best-effort 근실시간** vs deadline/스케줄링 관점 = 진짜 실시간. 이 대비가 우리 위치를 세움.
- 예상 결론 (교수님 예측): "스케줄링으로 끌고 가는 논문은 별로 없을 것, 대부분 '메모리 줄이고 실행시간 줄이고 정확도 유지' 논조" → 이걸 표로 **입증**하는 것이 서베이의 1차 목표.

### 산출물 2 — Elastic Scheduling 실전 응용 정리표

- 대상: elastic scheduling의 **practical 응용** — automotive, avionics 등 (이론 논문만이 아니라)
- 각 논문에서 정리할 것:

| 논문 | 응용 도메인 | **주된 가정** | 우리에게 적용 가능? | 가정이 깨지는 지점 |
| --- | --- | --- | --- | --- |
| (예) Buttazzo 2002 | 멀티미디어/제어 | C 고정, T만 가변 | 부분 | 우리는 C(W,M) 가변 |

- 목적: "기존 elastic scheduling의 전제와 우리 것이 부합하는지" 확인 → 부합하지 않는 지점이 곧 **확장 contribution** (피드백 8).

### 산출물 3 — 부하 설계 전략 (방향 1용)

- 유사 실시간성 평가 논문들(OSPERT 2024, MDPI ARM 등)이 부하를 **어떻게 설계했는지** 정리 → 우리 부하 조건을 근거 있게 확정.

### 산출물 4 — 고전 실시간 개념 노트

- 교수님이 언급한 계보: **imprecise computation** (mandatory/optional task), **mode change** (페이즈 전환, 태스크셋 교체), cyclic executive
- 각 개념이 우리 mode 전환과 어떻게 다르고 무엇을 빌려올 수 있는지 1페이지 노트.

### 진행 원칙 (기존 유지)
- 매 논문 2질문: ① 가변 변수가 무엇인가 ② 적응을 무엇이 트리거하나
- Claude Code로 paper card → comparison table 누적 (harness 활용)

## 5. 문제 정식화 — Schedulability Guarantee가 1번 질문 (피드백 4·5)

### 교수님의 핵심 질문 (그대로 기록)

> "이상 징후 시 정밀 모드로 하더라도 static하게 봤을 때는 항상 schedulable한가? **어떻게 guarantee 하지?** 어찌 보면 상충되는 거잖아."

→ 이것이 학위논문 문제 정식화의 **첫 번째 질문**. 기존 정리(W_phys ≤ W ≤ W_sys, utilization feasibility)는 뼈대로 유지하되, "보장 방법"을 명시해야 함.

### 보장 접근 후보 (서베이하며 확정)

1. **모든 mode가 개별적으로 schedulable하도록 mode bank를 설계** — 최정밀 mode조차 U ≤ U_lub이면 어떤 전환도 안전. 단, 그러면 정밀 mode의 정밀도에 상한이 생김 (상충의 정체).
2. **mode 전환 프로토콜**: 전환 순간의 과도 상태(transient overload)를 다루는 mode change protocol 계보 (Real-Time mode change 문헌) — 서베이 산출물 4와 연결.
3. **elastic 보장**: Buttazzo elastic은 압축 후 U ≤ U_lub을 유지하는 것이 정의 — 우리 확장(C 가변)에서도 같은 불변식이 유지됨을 보이는 방향.
- accuracy와 inference latency를 "어느 정도씩 모두 보장하는 값"이 존재하는가 — trade-off 곡선 위에서 feasible region을 정의하는 문제로 정식화.

### 즉시 실행 과제 — KCC 데이터로 U 계산 (피드백 5)

```
KCC 실측: (W, C) = (512, 40.3ms), (1024, 129.8ms), (2048, 460.3ms)
비중첩이므로 T = W/fs (fs=8kHz): 64ms, 128ms, 256ms

U(W) = C/T 계산:
  W=512:  40.3/64  = 0.63
  W=1024: 129.8/128 = 1.01  ← U > 1! deadline miss와 정합 (KCC에서 실제 miss)
  W=2048: 460.3/256 = 1.80  ← 심각한 초과

관찰: C가 O(W²)인데 T는 O(W)로만 늘어남
     → W를 늘리면 U가 오히려 증가 (U ∝ W)
     → W 축소는 latency만이 아니라 U 자체를 낮추는 레버
     → "정밀 mode로 갈수록 U가 커진다"는 상충 구조가 수치로 확인됨
```

- **max vs average**: 위는 max(WCET 기준). average C로도 계산해 두 곡선을 모두 제시 (교수님 지시). 실측 분포(σ, p99)가 이미 있으므로 바로 가능.
- 이 계산 결과가 "정밀 mode는 H를 함께 늘려야(주기 완화) feasible해진다"는 (W,H) 결합 조절의 필요성을 정량 입증 → 방향 2의 motivation 데이터.

## 6. Contribution 포지셔닝 (피드백 8 — 교수님 승인 프레임)

> "기존 elastic scheduling은 실행시간을 고정으로 가정하고 있다. 우리는 실행시간이 dynamic할 때도 쓸 수 있게 확장했다 — 이것도 contribution으로 쓸 수 있다. **novelty가 자연스럽게 확보되니까.**"

```
프레임: "새 기법 발명"이 아니라 "검증된 기법의 정직한 확장"

Buttazzo elastic:  τ_i = (C_i 고정, T_i ∈ [T_min, T_max], e_i)
본 연구 확장:      τ_diag = (C(W,M) 가변, T = H/fs 가변, 트리거 = q + slack)

확장 지점 3개:
  E1. 실행시간 축: C 고정 → C(W,M) — mode에 의해 결정되는 이산 가변
  E2. 트리거 축:  시스템 부하 단일 → 기계 상태(q) + slack 이중
  E3. 응용 축:    일반 태스크 → 진동 결함 진단 (accuracy가 U와 결합된 태스크)
```

- 서베이 산출물 2(기존 가정 정리표)가 이 확장의 근거 문서가 됨: "이 가정들이 우리 태스크에서 깨진다 → 그래서 확장한다"의 논리 사슬.

## 7. 투트랙 구도 (피드백 11)

| | 전자 (태훈) | 후자 (신영) |
| --- | --- | --- |
| 접근 | **RTOS + 스케줄링 정책**으로 문제 해결 | 시스템 SW 레벨 **경량화·최적화** (deterministic runtime) |
| 질문 | 정책만으로 가능한가, 정책+α가 필요한가 | 경량화만으로 정확도 손실 없이 가능한가 |
| 협력 | 신영의 경량화가 내 mode bank의 M축을 풍부하게 | 내 측정 체계·플랫폼이 신영의 평가 환경 |

- 극한 TinyML(초저사양 MCU) 방향은 신영 영역으로 정리 — 내 방향 1의 "상위 플랫폼 이전" 논리와 상보적.
- 상황 따라 서로 도움 주고받는 구조로 (교수님 제안).

## 8. 예상 질문 업데이트 (기존 7개 + 면담에서 나온 2개)

- **★ "이상 시 정밀 mode로 가도 schedulability를 어떻게 보장하나?"** → 5절의 접근: 모든 mode가 사전 admission 통과 + 전환 프로토콜. "보장 범위 안에서만 정밀해진다 = feasibility 내 최정밀"이 정책의 정의.
- **★ "모델이 바뀌어도 이 정책이 일반화되나?"** (교수님: "쉬우면서도 어려운 문제") → 정책은 C(W,M) 프로파일 테이블에만 의존하고 모델 내부를 보지 않음 — 어떤 모델이든 프로파일링만 하면 동작. 단, C(W) 곡선의 모양(O(W²)냐 선형이냐)은 모델마다 달라 mode 설계가 달라짐 — "정책은 일반, mode bank는 모델별" 이 두 층을 분리해 답변. FRFconv 외 1~2개 모델로 곡선이 다름을 실측해두면 방어가 완성됨 (방향 1 실험에 포함).
- 기존 7개 유지: 복잡성(mode 3~5개 룩업), 학습 기반 아님, mode 수, W-M 결합(모델 뱅크), online/offline(결정 시점), H만으론 안 되는 이유(O(W²)), anomaly score 출처.

## 부록 (0702~0707 정리, 유효 — 상세는 이전 절 그대로)

- A. 기호 정의: W[samples], H[samples], T=H/fs[s], C(W,M), U=C·fs/H, q, S / implicit deadline(D=T), fs 고정 가정
- B. mode = (W,H,M), (W,M)별 사전 학습 인스턴스 = model bank (image resizing 계보 방식)
- C. Online/Offline: 준비는 offline, 결정은 online — 기준은 "결정 시점" (DVFS·ABR 동일 구조)
- D. Anomaly score: softmax(초기) → 계층 트리거(+통계량) → SVDD(최종, 과제 연계)
- E. (q,S) 사분면 시나리오 5종: ①정상+부하→최경량 ②정상+여유→학습 양보 ③이상+여유→검증 후 최정밀 ④이상+포화→3단 압축(타 태스크→학습 중단→자기 타협) ⑤복귀→히스테리시스
